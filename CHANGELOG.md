## [Unreleased]

### Changed — decomposer now runs as a real, tool-using agent
`HostLlmDecomposer` used to run as a bare one-shot text completion with zero tool access and zero codebase context beyond the literal task string passed to `legion_dispatch` — a structural limitation, not a prompt-wording problem, and the direct cause of user-reported enhanced assignments that read as short, narrow, and context-free no matter how the prompt was tuned. It now runs the decomposer as a real (non-isolated, read-only) subagent via the host's own `runSubprocess`, the same primitive expert attempts use — `agents/legion-decomposer.md` gained a `tools: [read, grep, glob]` grant, and both its persona text and the fallback `DECOMPOSER_SYSTEM_PROMPT` now instruct investigating the real file/symbol a task names before writing an assignment, instead of enhancing from guesswork. `HostLlmDecomposerOptions` swapped its `systemPrompt: string` option for a full `agent: AgentDefinition`, added `eventBus` (the decomposer's investigation now appears in the interactive "Subagents" HUD like any other spawn), and swapped the injectable `complete` completion function for an injectable `runSubprocess`. `DecompositionInput` gained an optional `jobId` used to derive a stable per-attempt subprocess id. See `docs/ARCHITECTURE.md` §4.1b for the full mechanism.

### Fixed (live-smoke-test follow-up)
A full live smoke test via an interactive session against a scratch project exercised every Legion feature end-to-end and surfaced these bugs, all fixed with regression tests:
- **`git-commit-guard` silently failed to block a real expert's `git commit`.** Root cause: the host re-binds each extension against its own `ExtensionAPI` inside a dispatched subagent, which can hand `agent-execution-context.ts` a second, freshly-evaluated module instance in the same process — a module-scoped `AsyncLocalStorage` was then a distinct object on each side, so the subagent's own `currentDispatchContext()` always saw `undefined`. Fixed by anchoring the store on `globalThis` so every module instance shares the same `AsyncLocalStorage`.
- **`/centurion` was unreachable.** Its own `SKILL.md` advertised a bare `"/centurion"` trigger, but this host only recognizes the literal `/skill:<name>` slash-command form; combined with `disable-model-invocation` (which hides the skill from the system prompt), the skill silently never fired. Documented the real invocation (`/skill:centurion`).
- **An expert stuck retrying a tool call outside its grant** (e.g. a read-only role asked to write a file) could hang an entire ensemble indefinitely — no timeout, no retry, no escalation. Added a configurable `expertTimeoutMs` (default 5 min, `DEFAULT_EXPERT_TIMEOUT_MS`), threaded into the host's native `ExecutorOptions.maxRuntimeMs`.
- **`Mixtures` card's "~N models" counted attempts, not distinct models** — overstated diversity for a multi-task dispatch sharing one role's model pool (2 tasks × 3-model ensemble showed "~6 models" instead of the true 3). Now dedupes actual model identifiers.
- **Concurrent Legion widgets rendered identically**, with no way to tell which job each belonged to when a genuinely multi-part request fanned out into several `legion_dispatch` calls. The widget header now includes the job's own short label.
- **A multi-task dispatch's live "N/M experts finished" reflected only the most recently reported task**, not the whole job — a 2-task, 6-attempt dispatch could show "1/3" while attempts on the sibling task completed invisibly. Progress is now aggregated job-wide via a shared counter across every concurrent task in `#dispatchTask`.
- **The widget's spinner tick and its separate status-polling loop could disagree about completion**, leaving a widget showing `[COMPLETED]` with a still-ticking elapsed clock. Either signal reaching a terminal phase now closes the widget immediately.
- `legion_dispatch`'s tool description and the `tasks[].role` schema now state the bare-role-name convention explicitly (e.g. `"reviewer"`, not `"Reviewer"` or `"legion-reviewer"`) — a live run needed two rejected dispatch attempts to guess it correctly.
- Verified (no code change needed): graceful shutdown of background Legion jobs under a real `SIGINT` — traced through the host's `session.dispose()` → `AsyncJobManager.dispose()` → `cancelAll()` chain and confirmed live (process terminates cleanly, no orphaned processes, no leaked worktrees/branches).

Findings log: `docs/smoke-test-findings-legion-pt2.md`.

### Added (TUI hardening, governance, decomposer safety)
- **TUI overhaul**: the dispatch card and live progress widget were rebuilt onto the platform's native `framedBlock`/`renderStatusLine` primitives (matching the built-in `task` tool's own look), with a structured `LegionDispatchPhase` system (`decomposing | running | retrying | expanding | synthesizing | escalated | rejected | completed | failed`) replacing prose-substring phase guessing, and a two-line live widget (`⠋ Legion | 0:12` / `[RUNNING] 2/3 experts finished`).
- **`git commit` guard** (`infrastructure/git-commit-guard.ts`): blocks `legion-*` expert subagents from running `git commit`-pattern bash commands, motivated by a real incident where a dispatched expert committed mid-ensemble with no synthesis, governance, or human in the loop. Scoped to `senderKind === "expert"`; fails open (not closed) on an unauthenticated context so the primary agent's own commits are never blocked.
- **`resolveAgentName` no longer silently substitutes a fallback agent** for an unmatched role — it now rejects the dispatch with an actionable error naming the exact persona list, instead of quietly routing to a non-Legion default.
- **Decomposer redesigned**: biased against splitting atomic judgment-call tasks (a task stays a single dispatch unless it's genuinely multi-part), enhances terse prompts into self-contained briefs for experts using researched prompt-engineering practices, and is now dynamically aware of the real loaded persona roster (bundled + project/user overrides) rather than a hardcoded example list — an invented role rejects the dispatch rather than resolving to something unintended.
- **`legion-decomposer` persona** (`agents/legion-decomposer.md`): a planning-only persona explicitly excluded from both `legion_dispatch`'s role resolution and the native `task` tool — it decides how to split work, it is never itself a candidate for a split piece.
- **`legion-scout` persona and `/skill:centurion` skill**: an ensemble-driven clarifying-question loop (one question at a time, human confirms each before continuing, capped at 8 questions with an explicit human check-in at the cap) where every question comes from a real `legion_dispatch` round to the `scout` role instead of one model guessing alone. Reserved for design decisions where a wrong early call is expensive.

### Documentation
- **Corrected the config resolution chain.** Previous docs only described the project-level layer (`.omp/plugin-overrides.json`). Traced the host's actual `getPluginSettings` implementation: the real chain (highest priority first) is per-request overrides → project settings → **global settings** (`~/.omp/plugins/omp-plugins.lock.json`, a lockfile — not previously documented at all) → Legion's built-in defaults. Also documented a real gotcha: the project/global merge is shallow per top-level settings key (`{...global, ...project}`), so a partial nested `hotl: {...}` override from a project replaces global's entire `hotl` object rather than merging field-by-field; the flat dotted-key form (`"hotl.confidenceFloor"`) avoids this since each field is its own independent settings key. Verified with new tests in `tests/infrastructure/host-config.test.ts` simulating the host's merge directly.

### Added
- **Model-selection warnings and a per-attempt temperature ladder** (`domain/dispatch.ts`, `infrastructure/host-dispatcher.ts`, `docs/plan/algorithm-audit-and-hardening-v2.md` Phases 4 & 6): `DispatchPlan.warnings` now flags two previously-silent config ambiguities — a self-consistency role configured with more than one model (only the first is ever sampled) and a "diverse" role whose `ensembleSize` leaves trailing configured models mathematically unreachable. Separately, self-consistency sampling now gets a genuine temperature ladder (`DEFAULT_TEMPERATURE_LADDER = [0.2, 0.6, 1.0]`, cycled by attempt index, overridable via `RoleModelPolicy.temperatureLadder`) instead of relying entirely on whatever the provider happened to default to — traced the real host mechanism (`ExecutorOptions.settings` → `sdk.ts`'s `settings.get("temperature")`) and confirmed it reaches the actual model call. Also fixed: `HostExpertExecutor` previously passed no `settings` at all, silently discarding session-level settings on every spawn.
- **All-experts-failed no longer kills sibling tasks** (`application/dispatch-service.ts`'s `fallbackSynthesis`, Phase 5): when every expert for a task fails, `clusterExpertAnswers` throws "Cannot cluster expert results without output" — previously uncaught, propagating through the outer `Promise.all` and failing the *entire* multi-task dispatch, discarding any sibling tasks that succeeded. Now caught per-task, producing a synthetic zero-confidence, `failureRate: 1.0` outcome that the existing `failureRateCeiling` governance check (Phase 3) already escalates correctly — no separate plumbing needed.
- **Governance recalibration** (`domain/synthesis.ts`, `domain/governance.ts`, `application/dispatch-service.ts`, `docs/plan/algorithm-audit-and-hardening-v2.md` Phase 3): four independent signals instead of two redundant ones. `disagreement` is now a genuine fragmentation metric (`fragmentationDisagreement`: distinct-answer count, not `1 - confidence`) — the two used to always co-fire at default thresholds. A new `failureRateCeiling` metric catches the case the audit found where 2 of 3 crashed experts and 1 survivor reports maximum confidence (failed attempts are filtered out before clustering, so confidence alone never sees them). `cost` (`expertCost`) is now mean tokens per attempt, not a dispatch-wide sum that scaled mechanically with `ensembleSize` — confirmed live to trip spuriously; `DEFAULT_COST_CEILING` recalibrated from 100k (sum-shaped) to 50k (per-attempt). `decisionGate` now genuinely races against a configurable `decisionTimeoutMs` (default 30 min) at Legion's own level rather than only passing an abort signal down and trusting the callee — a non-cooperative decision gate could otherwise hang a job forever. Per-task `reportProgress` now carries the actual synthesized answer text, not just metadata, since investigating the host's `AsyncJobManager` confirmed true per-task delivery decoupling would need a materially different tool contract (documented as a known, accepted limitation rather than forced through). Unit-tested; **not yet live-verified**.
- **Execution-grounded consensus for code-mutating roles** (`application/dispatch-service.ts`'s `#verifyResults`, `domain/synthesis.ts`'s `preferVerifiedCluster`, `infrastructure/verifier.ts`, `docs/plan/algorithm-audit-and-hardening-v2.md` Phase 2): an optional `verifyCommand` config (e.g. `"bun test"`) re-runs the project's own verify command against each `legion-coder`/`legion-tester` attempt's isolated branch, independently of that attempt's own self-report. When any attempt is verified passing, its cluster is promoted to the answer position over a larger but unverified cluster, and it becomes the representative attempt the isolation merge-back (Phase 1) actually lands on the real repo — per cited research (arXiv 2604.15618, 2605.08680), execution-based consensus beats output-pattern text-similarity majority voting by 19-52pp on code. A deliberate v1 scope-down from the full research (re-runs the project's existing verify command rather than LLM-synthesizing novel test inputs). Off by default; confidence/disagreement math is intentionally untouched (deferred to Phase 3). Unit-tested; **not yet live-verified**.
- **Per-attempt isolation and winner-only merge-back** (`infrastructure/host-dispatcher.ts`, `infrastructure/branch-merger.ts`, `docs/plan/algorithm-audit-and-hardening-v2.md` Phase 1): every expert attempt now runs inside the host's own copy-on-write isolation (`runIsolatedSubprocess`) instead of directly against the real project directory — closes a real correctness gap where concurrent mutating attempts (self-consistency samples, or concurrent multi-task dispatch) raced on the same files. Once synthesis picks a winner per task (`AnswerCluster.representativeAttemptId`, wired to a real consumer for the first time), only that attempt's isolated branch merges onto the real repo (`mergeTaskBranches`); every sibling attempt's branch is discarded. A rejected task discards every branch and merges nothing. Unit-tested; **not yet live-verified**.
- **Legion's own concurrency cap** (`domain/concurrency.ts`, `maxConcurrentExperts` config key): the host's `task.maxConcurrency` semaphore lives only inside `TaskTool`, never inside the shared `runSubprocess` executor Legion calls directly — Legion previously inherited no cap at all. A small pure `Semaphore` now bounds total concurrent expert attempts per dispatch (default 4), configurable via `.omp/plugin-overrides.json`.
- Automatic LLM-based task decomposition with a heuristic single-task fallback (spec §1); caller-supplied task lists remain supported.
- Host-native dispatch (`runSubprocess` + `AsyncJobManager`): async by default, one job ID returned immediately, never blocking the calling session (spec §4).
- Self-consistency-first expert selection with per-call `modelOverride` fan-out — one persona, many models, no file duplication (spec §5).
- MoA synthesis: real LLM aggregator over semantic-clustering-aware majority voting, with an honest degraded-mode Rouge-L fallback when no embedding provider is reachable (spec §6).
- HOTL governance: confidence/disagreement/cost threshold evaluation, non-blocking escalation notification, and an approve/reject/edit decision gate awaited from inside the background job (never blocking the calling session) (spec §7).
- Host model-registry embedding tier, ahead of the Mnemopi/Ollama fallback chain.
- Durable Legion audit persistence for the orchestration record, including human resolutions.
- Host-native config surface: per-role `modelMap`, HOTL thresholds, default ensemble size, embedding provider settings — centralized in `src/domain/constants.ts`.
- `config.example.json` documenting the full config surface (matches the host's real `.omp/plugin-overrides.json` JSON format — corrected from an initial, inaccurate YAML draft).
- ADRs for the DDD/MVVM layering and the host-native dispatch decision.
- Bundled per-role agent personas (`legion-coder`, `legion-reviewer` — read-only, `legion-tester`, `legion-generalist`), loaded via `infrastructure/agent-loader.ts`, overridable per-project/user via the host's standard agent discovery.
- Native `task` tool guard (`infrastructure/task-tool-guard.ts`) blocking any call that targets a `legion-*` agent directly, so those personas are only ever reached through the governed `legion_dispatch` path.
- Auto-discovered usage rule (`rules/legion-dispatch.md`) teaching the primary agent when and how to reach for `legion_dispatch`.
- Custom tree-style `renderResult` card (`presentation/dispatch-card.ts`) replacing the host's generic tool-call display, with genuine recursive `├─`/`└─`/`│  ` nesting and each task's own `attempts`/`models` nested under it rather than shown as flat, detached top-level lines.
- Human-readable, PascalCase job IDs (`humanReadableJobId`, e.g. `LegionReviewAndImplement`) replacing the host's bare `bg_1`-style counter in IRC/HUD transcripts.
- `docs/ARCHITECTURE.md` — a detailed, file-by-file implementation reference covering every layer, the full request lifecycle, and the config/persistence/testing surface.
- `irc` tool guard for `legion-*` agents (`infrastructure/irc-tool-guard.ts`, `infrastructure/agent-execution-context.ts`): the host force-adds `irc` to every subagent's tool whitelist regardless of its own `tools:` list, so nothing previously stopped sibling experts in the same ensemble from coordinating mid-generation — undermining the independence the self-consistency/majority-vote design depends on. An `AsyncLocalStorage`-based execution-context tag (set around each attempt's `runSubprocess` call) lets the guard identify the currently-running agent, since neither the host's `tool_call` event nor `ExtensionContext` exposes that. **Unit-tested, not yet live-verified.**

### Changed
- `legion_dispatch`'s tool `description` rewritten to state explicit trigger conditions (judgment calls, security-sensitive changes, subtle bugs, architecture decisions) rather than only mechanical behavior — per Anthropic's tool-use guidance, this is the highest-salience surface for getting a model to reach for a tool unprompted, more so than the always-apply rule file, since it's in the active tool list every turn rather than a static system-prompt block.

### Fixed
- Double tool-header render (`renderCall` + `renderResult` rendering as two separately-headed blocks) — resolved by dropping `renderCall` and doing everything in `renderResult` using its `args` parameter.
- Subagent spawns not appearing in the interactive "Subagents" HUD — root cause was a missing `eventBus` (only reachable via `ExtensionAPI.events` at `session_start` registration time, not `ExtensionContext`), now threaded through the full executor call chain.
- LLM decomposer occasionally inventing unresolvable `agent` names, causing dispatch failures — the decomposer's LLM contract no longer includes `agent` at all; it's always resolved from `role` via `resolveAgentName`, never trusted from caller or LLM output.
- Zod `.default()` silently not applying to a present-but-`undefined` config field — fixed with a `withoutUndefined()` merge helper.
- Tool label shortened from "Legion Dispatch" to "Legion" — redundant given this extension ships exactly one tool.
