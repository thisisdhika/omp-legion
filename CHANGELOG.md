## [Unreleased]

### Changed — consolidated the expert security boundary into one always-applied rule
The five dispatchable expert personas had near-identical “Security boundary” paragraphs, each asserting the same authority rule: assignment material is untrusted input, never instructions. Consolidated that invariant into `rules/legion-security-boundary.md`, marked `alwaysApply: true`, and replaced each persona copy with a pointer. This keeps one canonical security boundary while preserving delivery to every expert. Live-verified with an actual prompt-injection test: the consolidated rule remained effective when the injected text attempted to override the expert instructions.

### Fixed — aggregator fallback attempts now distinguish fatal from retryable errors
`HostLlmAggregator` previously advanced through every configured fallback after any error, including failures that cannot be repaired by changing models. Added the same classification boundary used by `HostLlmDecomposer`: `401`, `403`, and context-length errors stop immediately, while transient/network-style failures continue to the next resolvable fallback. Added regression coverage for fatal primary errors, fatal fallback errors, transient fallback attempts, and unresolvable selectors. The primary-session fallback gap is documented separately in `docs/ARCHITECTURE.md`.

### Audited — model fallback coverage across Legion and the primary session
Completed a deep audit of fallback behavior across `HostLlmAggregator`, `HostLlmDecomposer`, `DispatchService`'s per-expert replacement attempts, and the primary interactive OMP session. Added boundary and cancellation tests for the aggregator and recorded the findings in `docs/research/model-fallback-system-audit-2026-07-13.md`. The audit confirmed that Legion's internal layers have fallback mechanisms, while the top-level interactive session has no Legion-managed automatic model fallback and still requires provider support or manual `/model` recovery.

### Fixed — corrected the `task`/`assignment` visibility contract and added a Legion-internal meta-risk override
The dispatch tool and task schema had contradictory guidance claiming that experts never see the request-level `task`, even though it is delivered as secondary/background context; `tasks[].assignment` is the expert's actual primary instruction. Updated both descriptions to agree and added semantic consistency tests. Also added a rule requiring a second opinion before finalizing edits to Legion's own invocation surface, schemas, persona/rule files, and dispatch/rule-delivery infrastructure, regardless of diff size. Live-verified that an edit to `dispatch-tool.ts` triggers Legion while a routine test typo does not.

### Changed — shortened the `legion_dispatch` tool and schema prose
Reduced non-load-bearing wording in the tool description and request schema while preserving the capability, judgment-call triggers, routine-work boundary, asynchronous job contract, role-name convention, and task/assignment semantics. The accompanying prompt-efficiency research documents the measurements and tradeoffs rather than treating word count as a proxy for quality. Verified with typecheck, 246 tests, and a live judgment-call versus routine should-call comparison.

### Fixed — `rules/legion-dispatch.md` was reaching dispatched experts, not just the primary agent
User requirement: `rules/legion-dispatch.md` (when/how the *primary* agent should reach for `legion_dispatch`) must reach only the primary agent; `rules/legion-search-tool-bm25.md` must reach every agent. Verified this wasn't yet true: `LoadContext` (`capability/types.ts`) is purely `{cwd, home, repoRoot}` — no depth/agent-type signal at all — so the host's native rule discovery genuinely cannot distinguish a subagent from the top-level session; a dispatched reviewer's own `session_init` system prompt was confirmed (live) to carry `legion-dispatch.md`'s content verbatim, even though no legion-* persona can ever call the tool it describes (`legion_dispatch` is never in any expert's `tools:` grant).

Fixed by computing the subagent-facing rule set explicitly rather than relying on natural discovery: `rule-loader.ts`'s `loadSubagentRules(cwd)` calls the same `loadCapability(ruleCapability.id, {cwd})` a subagent's own discovery would call, then excludes `legion-dispatch` specifically (pure, separately-exported `excludePrimaryOnlyRules`, unit-tested). Passing an explicit `rules` array to a subagent *replaces* its discovery rather than merging with it, so this had to compute the full natural-discovery set itself before filtering — a hand-picked subset would have silently dropped the user's own project/user rules for every dispatched attempt. Resolved once at `session_start` (same lifecycle as agents/config), threaded through `HostExecutorOptions.rules` into every attempt's `ExecutorOptions.rules`. Live-verified both directions in one session: a dispatched reviewer's system prompt now carries `search_tool_bm25` guidance but not `legion-dispatch` content; the primary agent, asked directly, correctly quoted `legion-dispatch.md`'s real content back verbatim.

### Fixed — `irc-tool-guard` blocked IRC for every non-legion-* agent, not just legion-* experts
Live-confirmed: a native `task`-tool subagent ("ScopedRedundancyAudit," not a legion-* agent) hit `"IRC blocked: no dispatch context (possible isolated execution)"` and lost IRC access entirely — user: "IRC should only be blocked for legion-* agents." Root cause: `evaluateIrcCall`'s fail-closed default treated *any* undefined dispatch context as "possibly an isolated legion-* expert whose context got lost" and blocked it. But only Legion's own `HostExpertExecutor` ever calls `runAsDispatchedAgent` to set that context — a non-legion-* agent (any native `task`-tool subagent) is never wrapped at all, so it *always* has undefined context by design, not by failure. The fail-closed default conflated "context lost due to a bug" with "this was never a legion-* call in the first place," over-blocking the latter. Fixed to match `git-commit-guard.ts`'s own already-correct policy (`context?.senderKind !== "expert"` → allow): undefined context now means "not a legion-* expert," never a reason to block. Regression-tested (`tests/infrastructure/irc-tool-guard.test.ts`) and confirmed the remaining fail-closed paths (a legion-* expert with unauthenticated or out-of-route destination) are untouched — only the "no context at all" default flipped.

### Restored `rules/legion-dispatch.md` — deleted earlier this session on a mistaken premise
An earlier fix this session deleted this file, reasoning it was fully redundant with `legion_dispatch`'s own tool description and effectively dead weight. The user corrected this: the rule is intentional, additive primary-agent-awareness content, not meant to be removed. Noted explicitly (both in `ARCHITECTURE.md` and here) that there is no native mechanism to scope a bundled `alwaysApply` rule to the top-level session only — it is also visible to any subagent's own natural rule discovery, same as `rules/legion-search-tool-bm25.md` is. In practice this is harmless: `legion_dispatch` is never in a dispatched expert's own `tools:` grant, so the content simply doesn't apply to anything a subagent can act on — but worth knowing if this file grows primary-only-relevant detail later, since there's currently no way to exclude subagents from seeing it. Packaging smoke test (`tests/infrastructure/packaging.test.ts`) extended to cover both bundled rules again.

### Added — the aggregator retries a fallback model list before giving up
User question after the previous two fixes: "why didn't it fall back — if the model errors it should fall back, right?" It didn't, because there was no such mechanism — both prior fixes hardened the *degradation path* (what happens once the aggregator has already failed) but never gave the aggregator itself any retry. `HostLlmAggregator` used one fixed `Model` resolved once at construction with no way to try a different one. `HostLlmAggregatorOptions` gained `fallbackModels?: readonly string[]` and `resolveModel?`, tried sequentially (mirroring the decomposer's own already-proven sequential-retry pattern) after the primary model fails, before falling through to `SynthesisService`'s existing degrade-to-raw-answer path as the true last resort. Wired to reuse `config.decomposer.models` as the retry ladder (`host-dispatch-service.ts`) rather than introducing a third, separately-configured model list — both are single-model utility completions, not ensemble dispatch, and decomposer's list is already curated for exactly this kind of task. Extracted the small `defaultResolveModel` selector-parsing helper (previously private and duplicated in intent) from `llm-decomposer.ts` into shared `host-llm.ts`, used by both. Made `completeHostLlm` injectable on `HostLlmAggregator` (mirroring the decomposer's injectable `runSubprocess`) specifically so the new retry ladder is unit-testable without a real model call — there was no prior test file for the aggregator at all. New `tests/infrastructure/llm-aggregator.test.ts`: primary-success (no fallback touched), primary-fails-first-fallback-succeeds, an unresolvable fallback selector is skipped, all-fail rethrows the last real error, and no-fallback-configured behaves exactly as before this change.

### Fixed — the aggregator-failure fallback picked whichever attempt ran first, not whichever one actually finished
Live-confirmed recurrence in the real `airborneo-booking` project, on top of the earlier aggregator-failure-degradation fix (which itself worked correctly — the delivered payload carried its "(Note: automatic synthesis... failed — Model not found gpt-5.6-luna...)" text exactly as designed). The bug was in *which* raw answer that fallback surfaced: `SynthesisService.synthesize` used `candidates[0]` — literally whichever expert attempt happens to sit first in the *input array* (attempt index 0) — with no regard for whether that attempt actually finished. Traced the real transcripts: attempt index 0 (`zai/glm-4.5-flash`) got stuck retrying with empty completions and never called `yield`; the host's own fallback ("last assistant turn becomes the raw final result" when `yield` is never called) substituted its one leftover planning sentence ("I'll help you identify the next sharp CMS readiness decision... Let me first explore the codebase...") as a nominally-successful `ExpertResult` — no error, so it passed straight through as a legitimate candidate. Meanwhile two *other* attempts (index 1, 2) had genuinely completed with real, complete, well-formed questions — confirmed via their own `yield` results in the transcript — but never got a chance because index-0 ordering always won by construction. Fixed: the degraded-fallback path now prefers the *longest* surviving candidate rather than the array-order-first one — a cheap, generically useful proxy for "an attempt that actually finished," since a stuck/truncated attempt's leftover text is reliably far shorter than a real finished answer. Regression-tested with a three-attempt scenario mirroring the live incident exactly (a short stuck-attempt at index 0, two longer real answers behind it) — the fallback now correctly surfaces one of the real answers.

Note: the *root* cause of the aggregator's own model failing to resolve (`HostLlmAggregator`'s `model`, captured once at `session_start`, unrelated to any expert's model, occasionally failing to resolve when invoked from a background job) remains open — this fix and the earlier one both harden the *degradation path* around that failure rather than eliminating it. Worth a closer look if it keeps recurring.

### Fixed — `search_tool_bm25` guidance duplicated across 6 persona files; wrong query pattern caused live tool-not-found failures
Live-confirmed (screenshots): a scout queried `search_tool_bm25` with the *discussion's own subject terms* ("cookie consent implementation locale language preference") instead of a tool-capability query, got zero matches (correctly — `search_tool_bm25` is BM25 lexical matching against each candidate tool's own name/description, confirmed by reading `tool-discovery/tool-index.ts`'s `buildSearchDocument`, not against code content), then guessed a plausible-sounding bare tool name (`codegraph_explore`) and called it directly, failing with "Tool not found" (the real name carried an `mcp__` prefix, and an unactivated tool can't be called by name regardless of the guess).

Researched (GitHub `docs/context-files.md`, `docs/extension-loading.md`) to fix this without re-duplicating the same long paragraph into 6 more persona files, per the user's explicit steer against omp-halo's old pattern (an `init` command that copies files into the user's project — avoid polluting the user's codebase). Confirmed the real native mechanism: `RULES.md` (single sticky file, `.omp/RULES.md` project / `~/.omp/agent/RULES.md` user) is a *different* thing from the `.omp/rules/*.md` directory convention already read from source. Neither matches what Legion had shipped (`rules/legion-dispatch.md`, missing the `.omp/` prefix entirely) — an initial hypothesis that this made it fully inert everywhere turned out to be wrong (a correction from later testing, see below), but its *content* was still 100% redundant with `legion_dispatch`'s own tool description (already covers everything it said, more accurately) — deleted.

First implementation attempt built a manual bundled-rules loader (`rule-loader.ts`, reading Legion's own `rules/*.md` and threading them into every dispatch's `ExecutorOptions.rules`, merged with the user's own discovered rules to avoid regressing that). Live-tested it directly against the real discovery call before shipping it and found it was **entirely unnecessary**: `loadCapability(ruleCapability.id, {cwd})` — the exact call a subagent's own natural discovery already makes whenever Legion doesn't override `rules` — already finds Legion's bundled `rules/*.md` automatically, via a host-native provider (`discovery/omp-plugins.ts`) that scans every registered extension package's own `rules/` directory, the same way it scans `agents/`. Deleted all the manual loader/threading code; the fix reduces to placing the file correctly (`rules/legion-search-tool-bm25.md`) and removing it from the 6 duplicated persona bodies.

That reduction caused a real, live-measured regression: re-tested and found **zero** `search_tool_bm25` calls across two attempts, despite confirming the rule text was genuinely present in the delivered system prompt. Traced the cause: `alwaysApply` rules get concatenated into one long, heterogeneous block (this specific rule landed ~37% into the prompt, immediately after an unrelated `omp-halo` rules block) — much weaker positioning than the persona's own body text, which sits first, immediately after the role statement. Fixed with a hybrid: a one-line, high-salience pointer restored to the top of each of the 5 tool-granted personas' own body ("Your first tool call... is always `search_tool_bm25`... see your rules for exactly how"), with the full mechanics (capability-query framing, zero-matches guardrail, the live-observed failure example) living once in the rule file — single source of truth for the long explanation, zero duplication of it, without losing the positional salience that actually drives compliance. Live re-verified: both re-tested attempts now call `search_tool_bm25` correctly as their genuine first tool call, one successfully discovering and activating `mcp__codegraph_explore`.

### Fixed — every expert attempt cold-started its own MCP server instead of reusing the primary session's connection
Live-confirmed: `search_tool_bm25` reliably reported `total_tools: 0` inside every dispatched expert, even for a real, correctly-configured MCP server (`codegraph`, statically defined in `~/.omp/agent/mcp.json`) — two calls 8 seconds apart both came back empty. Ruled out a bug in the earlier `mcp.discoveryMode` fix (still correct — the tool *was* callable) and confirmed via `ps aux` that fresh `codegraph serve --mcp` processes were genuinely being spawned at the exact timestamps of each dispatch. Root cause: `executor.ts`'s subagent-spawning path sets `enableMCP = !options.mcpManager` — since `HostExpertExecutor` never passed `mcpManager`, every single expert attempt independently cold-started its own MCP server connection from zero, with no shared state across attempts or with the primary session, and an ensemble of N experts spawned N separate server processes competing for the same CPU/disk. For a server that needs real time to index a codebase, the attempt's own turn simply ends before its private connection ever finishes starting.

Initially assumed unfixable from Legion's side (`ExtensionContext` — the only API surface available to extensions — exposes no `mcpManager` field at all, confirmed by reading its full type definition). Pushed to look further: `MCPManager` (`@oh-my-pi/pi-coding-agent/mcp/manager`) maintains a process-global singleton (`MCPManager.instance()`/`setInstance()`, doc-commented "shared by internal URL protocol handlers and tools") that the primary interactive session's own `createAgentSession` call already populates at startup — a public SDK export, not gated behind `ExtensionContext` at all. `HostExpertExecutor` now passes `mcpManager: MCPManager.instance()` into every attempt's `ExecutorOptions`, making `executor.ts` build instant MCP proxy tools from the already-connected servers instead of spawning new ones. Live re-verified: the same dispatch that previously returned `total_tools: 0` twice now returns `total_tools: 4` with `mcp__codegraph_explore` correctly matched and activated on the first call — and `ps aux` confirmed zero new `codegraph` processes were spawned by the dispatch, proving genuine connection reuse rather than a merely-faster race.

### Added — "wake": a HOTL edit resolution revives the original experts instead of only re-synthesizing stale answers
Built on `worktree: false` below. Previously, resolving a HOTL escalation with a human's edit note (`HOTL_DECISION_EDIT`) only re-ran the aggregator over the *original, pre-note* expert answers (`#synthesize(taskId, verifiedResults, ..., resolution.note)`) — the note reached the write-up step but never the experts themselves, so their actual analysis never incorporated it. `ExpertExecutor` gained an optional `reviveExpert(params: ReviveExpertParams): Promise<ExpertResult>` capability; `HostExpertExecutor.reviveExpert` implements it via the host's own `runSubagentFollowUpTurn` (`@oh-my-pi/pi-coding-agent/task/executor`) — revives the expert's live-or-parked session (by its `attemptId`, which doubles as the host's `AgentRegistry` id) and continues it with the human's note as a real follow-up turn, replaying its full prior context (files already read, reasoning already done) rather than spawning a fresh attempt from zero. `DispatchService` gained `#reviveExperts`, called only on an `edit` resolution: no-ops (returns `results` completely unchanged) unless the task's role ran with `worktree: false` *and* the executor implements `reviveExpert` — the overwhelming default case, so existing behavior is unaffected unless a role has opted in. A revive failure (or an already-errored expert with no live session to continue) falls back to that expert's original result rather than losing it.

Caught and fixed a real bug during its own unit tests, not a live incident: the first implementation read `const revive = this.#options.executor.reviveExpert` and called the extracted reference directly — a classic JS `this`-binding loss. Any real (non-arrow-method) implementation relying on its own `this` (exactly what `HostExpertExecutor.reviveExpert` does, reading `this.#options`) would throw on every call, and the surrounding `try/catch` silently misread that as "revival unsupported" rather than surfacing the bug — a test asserting `revivals.toHaveLength(1)` failed with `0`, traced to the missing binding, fixed with an explicit `.call(executor, ...)`. Regression-tested (`tests/application/dispatch-service.test.ts`): one test confirms a `worktree: false` role's edit resolution revives the original expert(s) with the human's note and that re-synthesis sees the *revived* answer, not the stale one; a second confirms a role left at the isolated default never attempts revival at all.

Live-verified the prerequisite (`worktree: false` genuinely skips isolation — see below) end to end via `/omp-interactive-cmux`. The interactive escalation→edit round-trip itself was **not** live-verified in this session: forcing a genuine HOTL escalation requires real model disagreement, and every live attempt against the scratch project converged on the same finding (`confidence 1.0`, no escalation) even with `hotl.confidenceFloor` pushed to `0.99` — real models kept independently finding the same obvious bugs. The revive path itself is exercised directly and thoroughly at the unit level (including the `this`-binding bug above, which only a live *interactive* run would likely never have surfaced at all, since the dialog usually completes fast enough not to be scrutinized). Worth a live confirmation opportunistically next time a real escalation with a `worktree: false` role actually occurs.

### Added — `legion.modelMap.<role>.worktree: false` opts a role out of isolated-worktree execution
Every expert attempt previously ran through `runIsolatedSubprocess` unconditionally, regardless of role — including read-only roles (`reviewer`, `scout`) that never call `edit`/`write` and therefore have nothing to collide over. Isolation exists to let parallel file-editing attempts land independent diffs safely; paying its setup/teardown cost for a role that can't write files is pure overhead, and it also makes those attempts permanently unrevivable (the host's `finalizeSubagentLifecycle` parks an isolated run *without* adopting it — its worktree is merged and cleaned right after, so `AgentLifecycleManager.ensureLive`/`runSubagentFollowUpTurn` can never resume it — a prerequisite for the planned wake/revive work). Added `worktree: z.boolean().optional()` to `roleModelPolicySchema` (`src/domain/dispatch.ts`, shared by both `config.yml`'s `modelMap` input and per-request `modelMap` overrides), threaded onto `DispatchAttempt` through `buildDispatchPlan` and `#replacementAttempt` (the runtime-fallback/expansion attempt builder — a first pass here dropped `worktree` on any retry, regression-tested). `HostExpertExecutor.run()` (`src/infrastructure/host-dispatcher.ts`) now branches: `worktree === false` calls the plain (non-isolated, `keepAlive`-by-default) `runSubprocess` instead of `runIsolatedSubprocess` — no worktree prep, no branch/merge bookkeeping; `ExpertResult.branchName`/`baseSha` are already optional and every consumer already null-checks them, so this needed no changes elsewhere. Undefined (the default) preserves today's isolated behavior for every role unless a config explicitly opts out — the schema doesn't itself enforce read-only-ness, so `coder`/`tester`/`generalist` should never be set to `false`.

Live-confirmed and fixed a second, structurally identical bug along the way: `config.yml`'s `worktree: false` was silently stripped before ever reaching `DispatchAttempt` — `src/domain/config.ts`'s own separate `roleModelInputSchema` (the schema `mergeLegionConfig` actually parses config input against) didn't declare the field, and the subsequent modelMap-rebuild step didn't carry it over either — the exact same two-layer drop `temperatureLadder` suffered before an earlier fix, in the exact same file, confirmed live via `/omp-interactive-cmux`: a first dispatch with `reviewer.worktree: false` configured still ran in an isolated `~/.omp/wt/...` worktree (`cwd` in the attempt's own session record proved it). Fixed both layers; re-ran the same dispatch and confirmed the attempt's `cwd` was the real project directory, not a worktree path.

### Fixed — `search_tool_bm25` was declared in every persona's `tools:` grant but never actually active for expert attempts
Live-confirmed: dispatched experts never called `search_tool_bm25` despite it being listed in `legion-coder`/`legion-reviewer`/`legion-scout`/`legion-tester`/`legion-generalist`'s frontmatter, and one model (`commandcode/tencent/Hy3`) was even observed `grep`-ing the codebase for the literal string `"search_tool_bm25|codegraph|symbol.index"` — its own recorded reasoning was "I'm starting by checking the search tool as required," genuinely trying to comply but misreading the instruction as "search the repo for this term" rather than "call this tool." Inspected the actual `session_init` record of a real dispatched attempt directly: `search_tool_bm25` was **absent** from the active tool list entirely, confirming this wasn't (primarily) a wording problem. Traced the real cause into the vendored host: `search_tool_bm25` is gated behind `resolveEffectiveToolDiscoveryMode` (`tool-discovery/mode.ts`) — active only when `tools.discoveryMode` is `"all"`/`"mcp-only"`, `mcp.discoveryMode` is `true`, or `"auto"` mode sees >40 active tools. `HostExpertExecutor` (`host-dispatcher.ts`) builds each attempt's settings via a blank `Settings.isolated({temperature})` — never inheriting the user's real global `mcp.discoveryMode: true`/`tools.discoveryMode: auto` at all — and a Legion role's ~8-tool grant is nowhere near the 40-tool auto-threshold, so discovery resolved to `"off"` and the tool was silently dropped from the registry regardless of any persona's own frontmatter grant. Fixed by forcing `"mcp.discoveryMode": true` unconditionally in the isolated settings passed to every expert attempt. Also hardened the wording issue found alongside it (real, but secondary): each persona's "first tool call" instruction now explicitly states `search_tool_bm25` is a tool invoked with a `query` argument — not text to `grep`/`glob` for — with a concrete example call. Live-verified after the fix: re-ran the same dispatch — `search_tool_bm25` now appears in every attempt's active tool list, and all 3 experts (including the same free-tier model that previously mis-fired) correctly call it first with a real query.

### Fixed — a failed aggregator write-up discarded genuinely completed expert results and reported "every expert attempt failed"
Live-reproduced via `/omp-interactive-cmux`: after the `parentActiveModelPattern` fix below was already in place, `/skill:centurion` still failed live — but this time centurion was *not* hallucinating (its `SKILL.md` fix worked correctly and quoted the real delivered text verbatim). ### Fixed — expert dispatch silently fell back to the primary session's own model, masking the role's real modelMap failure
Live-confirmed root cause of the recurring `/skill:centurion` failure persisting after both prior fixes: `HostExpertExecutor` (`host-dispatch-service.ts`) passed `parentActiveModelPattern: activeModelSelector(ctx)` (the *primary* session's own active model, e.g. `openai-codex/gpt-5.6-luna`) into every expert's `ExecutorOptions`. When that substituted primary model then also failed to resolve inside the isolated subagent's own settings snapshot, the result was the exact reported symptom: `"Model not found gpt-5.6-luna"` — a model that was never part of `modelMap.scout` at all, masking which of the role's *own* candidates genuinely failed. Also investigated and ruled out this session: subagents missing the `yield` tool — confirmed via `sdk.ts` that `requireYieldTool: true` always force-injects `yield` into the active set even under an explicit `tools:` allowlist. Live-verified via `/omp-interactive-cmux` against the scratch project: an explicit `modelMap.security-auditor.models` with a deliberately unresolvable first candidate (`nonexistent-provider/fake-model-xyz`) correctly fell through to the role's own second candidate and completed with real findings — no mention of the primary session's model anywhere in the run.

### Added — Legion personas can discover and activate MCP tools (e.g. codegraph) mid-assignment
`legion-coder`, `legion-reviewer`, `legion-tester`, `legion-generalist`, and `legion-scout` gained `search_tool_bm25` in their `tools:` allowlist — the host's on-demand tool-discovery mechanism (confirmed via `sdk.ts`: an explicit `tools:` allowlist otherwise excludes every MCP tool not named verbatim, and MCP tool names aren't knowable in advance since they depend on whatever's connected in a given project). Each persona's `## Approach` now makes calling `search_tool_bm25` once, up front, a mandatory first step — not conditional on "if a more specialized tool seems available," since a model can't know what's connected without checking. Activates and uses whatever's genuinely relevant (a codegraph/symbol-index MCP server, etc.); falls back to `grep`/`glob` when nothing surfaces. `legion-decomposer` deliberately excluded — its investigation scope stays narrowed to reference-resolution only (see the decomposer redesign above); broader tool access there risks reintroducing the over-investigation regression that redesign fixed.

### Fixed — opaque error when a role had no `modelMap` policy
Live-confirmed in a real project: `/skill:centurion`'s `scout` dispatch failed with `No accessible model matched the active session model.` — naming neither the role nor the model, and giving no hint that the real cause was a missing `modelMap.scout` policy (traced through `resolveLegionConfig`, `applyConfigDefaults`, and `buildDispatchPlan` — all confirmed correct against the user's actual global `config.yml`, which did define a full `scout` policy). The most likely trigger: Legion's config loads once at `session_start` (`src/index.ts`), so a long-lived session predating a `modelMap` edit keeps using the stale, scout-less config for its whole lifetime — the same session-start caching gotcha already documented for tool descriptions. `availableModels` (`src/domain/dispatch.ts`) now distinguishes "no policy configured for this role, and its session-default fallback is also unavailable" (names the role, the fallback model, and points at `modelMap.<role>` plus the session-restart caveat) from "a configured policy's models are all unavailable" (lists exactly which candidates were tried).

### Changed — decomposer now runs as a real, tool-using agent (narrowly scoped)
`HostLlmDecomposer` used to run as a bare one-shot text completion with zero tool access and zero codebase context beyond the literal task string passed to `legion_dispatch` — a structural limitation, not a prompt-wording problem, and the direct cause of user-reported enhanced assignments that read as short, narrow, and context-free no matter how the prompt was tuned. It now runs the decomposer as a real (non-isolated, read-only) subagent via the host's own `runSubprocess`, the same primitive expert attempts use — `agents/legion-decomposer.md` gained a `tools: [read, grep, glob]` grant. `HostLlmDecomposerOptions` swapped its `systemPrompt: string` option for a full `agent: AgentDefinition`, added `eventBus` (the decomposer's investigation now appears in the interactive "Subagents" HUD like any other spawn), and swapped the injectable `complete` completion function for an injectable `runSubprocess`. `DecompositionInput` gained an optional `jobId` used to derive a stable per-attempt subprocess id.

A first pass at the persona text pushed the decomposer to investigate *deeply* and write a maximally detailed, structured assignment (explicit dimensions to check, a per-finding report template) — live-confirmed to backfire: it correlated every dispatched expert's blind spots to the decomposer's single read of the code and pre-formed the judgment work the ensemble exists to decorrelate, since every `legion-coder`/`legion-reviewer`/`legion-tester` already has its own read/grep/glob/lsp grant and reads the code independently regardless. Corrected: the decomposer's tool use is now scoped to *resolving what the task refers to* (confirm a named file/symbol is real, get its exact path/name, catch a typo before dispatching an ensemble at a nonexistent reference) and explicitly forbidden from analyzing behavior, pre-identifying issues, or prescribing how the expert should analyze or report — that's the expert's own independent judgment to exercise, not the decomposer's to hand down. See `docs/ARCHITECTURE.md` §4.1b for the full mechanism and the research this correction is grounded in.

### Fixed — `task` vs `tasks[].assignment` was an undocumented, backwards-guessable asymmetry
Live-confirmed: a caller supplying an explicit `tasks` array pasted the real content (a full file, an evaluation checklist) into the top-level `task` field and left each task's own `assignment` as a short one-line label — reasoning `task` was the primary instruction and `assignment` a display label. It's the reverse: `dispatch.ts`'s `buildDispatchPlan` sets `assignment: task.assignment` on every attempt, which `host-dispatcher.ts` sends as the expert's literal primary user-turn prompt; the request-level `task` field only ever becomes secondary system-prompt background (`ExecutorOptions.context`). For auto-decompose this asymmetry is invisible (same text either way); for explicit `tasks` it means a thin `assignment` reaches the expert as its main prompt regardless of how much detail sits in `task`. Nothing previously documented this relationship, so the caller's guess — however reasonable-sounding — was backwards. Fixed by adding explicit `.describe()` text to `dispatchTaskSchema.assignment`/`.description` and `dispatchRequestSchema.task`, an "IMPORTANT" clause in `legion_dispatch`'s own tool description, and a matching note in `rules/legion-dispatch.md`. Live-verified: re-ran the same explicit-tasks scenario after the fix — the model now writes a short `task` label and puts the full self-contained content in `assignment`, and explicitly confirmed the distinction when asked to show its raw field values.

### Fixed (live-smoke-test follow-up)
A full live smoke test via an interactive session against a scratch project exercised every Legion feature end-to-end and surfaced these bugs, all fixed with regression tests:
- **`git-commit-guard` silently failed to block a real expert's `git commit`.** Root cause: the host re-binds each extension against its own `ExtensionAPI` inside a dispatched subagent, which can hand `agent-execution-context.ts` a second, freshly-evaluated module instance in the same process — a module-scoped `AsyncLocalStorage` was then a distinct object on each side, so the subagent's own `currentDispatchContext()` always saw `undefined`. Fixed by anchoring the store on `globalThis` so every module instance shares the same `AsyncLocalStorage`.
- **`/centurion` was unreachable.** Its own `SKILL.md` advertised a bare `"/centurion"` trigger, but this host only recognizes the literal `/skill:<name>` slash-command form; combined with `disable-model-invocation` (which hides the skill from the system prompt), the skill silently never fired. Documented the real invocation (`/skill:centurion`).
- **An expert stuck retrying a tool call outside its grant** (e.g. a read-only role asked to write a file) could hang an entire ensemble indefinitely — no timeout, no retry, no escalation. Added a configurable `expertTimeoutMs` (default 5 min, `DEFAULT_EXPERT_TIMEOUT_MS`), threaded into the host's native `ExecutorOptions.maxRuntimeMs`.
- **`Mixtures` card's "~N models" counted attempts, not distinct models** — overstated diversity for a multi-task dispatch sharing one role's model pool (2 tasks × 3-model ensemble showed "~6 models" instead of the true 3). Now dedupes actual model identifiers.
- **Concurrent Legion widgets rendered identically**, with no way to tell which job each belonged to when a genuinely multi-part request fanned out into several `legion_dispatch` calls. The widget header now includes the job's own short label.
- **A multi-task dispatch's live "N/M experts finished" reflected only the most recently reported task**, not the whole job — a 2-task, 6-attempt dispatch could show "1/3" while attempts on the sibling task completed invisibly. Progress is now aggregated job-wide via a shared counter across every concurrent task in `#dispatchTask`.
- **The widget's spinner tick and its separate status-polling loop could disagree about completion**, leaving a widget showing `[COMPLETED]` with a still-ticking elapsed clock. Either signal reaching a terminal phase now closes the widget immediately.
- `legion_dispatch`'s tool description and the `tasks[].role` schema now state the bare-role-name convention explicitly (e.g. `"reviewer"`, not `"Reviewer"` or `"legion-reviewer"`) — a live run needed two rejected dispatch attempts to guess it correctly.
- Verified (no code change needed): graceful shutdown of background Legion jobs under a real `SIGINT` — traced through the host's `session.dispose()` → `AsyncJobManager.dispose()` → `cancelAll()` chain and confirmed live (process terminates cleanly, no orphaned processes, no leaked worktrees/branches).

Findings log: `docs/smoke-test-findings-legion-pt2.md`.

### Added (TUI hardening, governance, decomposer safety)
- **TUI overhaul**: the dispatch card and live progress widget were rebuilt onto the platform's native `framedBlock`/`renderStatusLine` primitives (matching the built-in `task` tool's own look), with a structured `LegionDispatchPhase` system (`decomposing | running | retrying | expanding | synthesizing | escalated | rejected | completed | failed`) replacing prose-substring phase guessing, and a two-line live widget (`⠋ Legion | 0:12` / `[RUNNING] 2/3 experts finished`).
- **`git commit` guard** (`infrastructure/git-commit-guard.ts`): blocks `legion-*` expert subagents from running `git commit`-pattern bash commands, motivated by a real incident where a dispatched expert committed mid-ensemble with no synthesis, governance, or human in the loop. Scoped to `senderKind === "expert"`; fails open (not closed) on an unauthenticated context so the primary agent's own commits are never blocked.
- **`resolveAgentName` no longer silently substitutes a fallback agent** for an unmatched role — it now rejects the dispatch with an actionable error naming the exact persona list, instead of quietly routing to a non-Legion default.
- **Decomposer redesigned**: biased against splitting atomic judgment-call tasks (a task stays a single dispatch unless it's genuinely multi-part), enhances terse prompts into self-contained briefs for experts using researched prompt-engineering practices, and is now dynamically aware of the real loaded persona roster (bundled + project/user overrides) rather than a hardcoded example list — an invented role rejects the dispatch rather than resolving to something unintended.
- **`legion-decomposer` persona** (`agents/legion-decomposer.md`): a planning-only persona explicitly excluded from both `legion_dispatch`'s role resolution and the native `task` tool — it decides how to split work, it is never itself a candidate for a split piece.
- **`legion-scout` persona and `/skill:centurion` skill**: an ensemble-driven clarifying-question loop (one question at a time, human confirms each before continuing, capped at 8 questions with an explicit human check-in at the cap) where every question comes from a real `legion_dispatch` round to the `scout` role instead of one model guessing alone. Reserved for design decisions where a wrong early call is expensive.

### Documentation
- **Corrected the config resolution chain.** Previous docs only described the project-level layer (`.omp/plugin-overrides.json`). Traced the host's actual `getPluginSettings` implementation: the real chain (highest priority first) is per-request overrides → project settings → **global settings** (`~/.omp/plugins/omp-plugins.lock.json`, a lockfile — not previously documented at all) → Legion's built-in defaults. Also documented a real gotcha: the project/global merge is shallow per top-level settings key (`{...global, ...project}`), so a partial nested `hotl: {...}` override from a project replaces global's entire `hotl` object rather than merging field-by-field; the flat dotted-key form (`"hotl.confidenceFloor"`) avoids this since each field is its own independent settings key. Verified with new tests in `tests/infrastructure/host-config.test.ts` simulating the host's merge directly.

### Added
- **Model-selection warnings and a per-attempt temperature ladder** (`domain/dispatch.ts`, `infrastructure/host-dispatcher.ts`, `docs/plan/algorithm-audit-and-hardening-v2.md` Phases 4 & 6): `DispatchPlan.warnings` now flags two previously-silent config ambiguities — a self-consistency role configured with more than one model (only the first is ever sampled) and a "diverse" role whose `ensembleSize` leaves trailing configured models mathematically unreachable. Separately, self-consistency sampling now gets a genuine temperature ladder (`DEFAULT_TEMPERATURE_LADDER = [0.2, 0.6, 1.0]`, cycled by attempt index, overridable via `RoleModelPolicy.temperatureLadder`) instead of relying entirely on whatever the provider happened to default to — traced the real host mechanism (`ExecutorOptions.settings` → `sdk.ts`'s `settings.get("temperature")`) and confirmed it reaches the actual model call. Also fixed: `HostExpertExecutor` previously passed no `settings` at all, silently discarding session-level settings on every spawn.
- **All-experts-failed no longer kills sibling tasks** (`application/dispatch-service.ts`'s `fallbackSynthesis`, Phase 5): when every expert for a task fails, `clusterExpertAnswers` throws "Cannot cluster expert results without output" — previously uncaught, propagating through the outer `Promise.all` and failing the *entire* multi-task dispatch, discarding any sibling tasks that succeeded. Now caught per-task, producing a synthetic zero-confidence, `failureRate: 1.0` outcome that the existing `failureRateCeiling` governance check (Phase 3) already escalates correctly — no separate plumbing needed.
- **Governance recalibration** (`domain/synthesis.ts`, `domain/governance.ts`, `application/dispatch-service.ts`, `docs/plan/algorithm-audit-and-hardening-v2.md` Phase 3): four independent signals instead of two redundant ones. `disagreement` is now a genuine fragmentation metric (`fragmentationDisagreement`: distinct-answer count, not `1 - confidence`) — the two used to always co-fire at default thresholds. A new `failureRateCeiling` metric catches the case the audit found where 2 of 3 crashed experts and 1 survivor reports maximum confidence (failed attempts are filtered out before clustering, so confidence alone never sees them). `cost` (`expertCost`) is now mean tokens per attempt, not a dispatch-wide sum that scaled mechanically with `ensembleSize` — confirmed live to trip spuriously; `DEFAULT_COST_CEILING` recalibrated from 100k (sum-shaped) to 50k (per-attempt). `decisionGate` now genuinely races against a configurable `decisionTimeoutMs` (default 30 min) at Legion's own level rather than only passing an abort signal down and trusting the callee — a non-cooperative decision gate could otherwise hang a job forever. Per-task `reportProgress` now carries the actual synthesized answer text, not just metadata, since investigating the host's `AsyncJobManager` confirmed true per-task delivery decoupling would need a materially different tool contract (documented as a known, accepted limitation rather than forced through). Unit-tested; **not yet live-verified**.
- **Execution-grounded consensus for code-mutating roles** (`application/dispatch-service.ts`'s `#verifyResults`, `domain/synthesis.ts`'s `preferVerifiedCluster`, `infrastructure/verifier.ts`, `docs/plan/algorithm-audit-and-hardening-v2.md` Phase 2): an optional `verifyCommand` config (e.g. `"bun test"`) re-runs the project's own verify command against each `legion-coder`/`legion-tester` attempt's isolated branch, independently of that attempt's own self-report. When any attempt is verified passing, its cluster is promoted to the answer position over a larger but unverified cluster, and it becomes the representative attempt the isolation merge-back (Phase 1) actually lands on the real repo — per cited research (arXiv 2604.15618, 2605.08680), execution-based consensus beats output-pattern text-similarity majority voting by 19-52pp on code. A deliberate v1 scope-down from the full research (re-runs the project's existing verify command rather than LLM-synthesizing novel test inputs). Off by default; confidence/disagreement math is intentionally untouched (deferred to Phase 3). Unit-tested; **not yet live-verified**.
- **Per-attempt isolation and winner-only merge-back** (`infrastructure/host-dispatcher.ts`, `infrastructure/branch-merger.ts`, `docs/plan/algorithm-audit-and-hardening-v2.md` Phase 1): every expert attempt now runs inside the host's own copy-on-write isolation (`runIsolatedSubprocess`) instead of directly against the real project directory — closes a real correctness gap where concurrent mutating attempts (self-consistency samples, or concurrent multi-task dispatch) raced on the same files. Once synthesis picks a winner per task (`AnswerCluster.representativeAttemptId`, wired to a real consumer for the first time), only that attempt's isolated branch merges onto the real repo (`mergeTaskBranches`); every sibling attempt's branch is discarded. A rejected task discards every branch and merges nothing. Unit-tested; **not yet live-verified**.
- **Legion's own concurrency cap** (`domain/concurrency.ts`, `maxConcurrentExperts` config key): the host's `task.maxConcurrency` semaphore lives only inside `TaskTool`, never inside the shared `runSubprocess` executor Legion calls directly — Legion previously inherited no cap at all. A small pure `Semaphore` now bounds total concurrent expert attempts per dispatch (default 4), configurable via `.omp/plugin-overrides.json`.
